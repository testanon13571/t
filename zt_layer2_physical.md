## Layer 2: Physical & Peripheral

**CISA Pillars:** Devices, Data (contributing)

### Threats Addressed

Physical and peripheral threats target the endpoint hardware directly, bypassing OS-layer security controls by attacking the device before the OS loads or through physical interfaces that the OS cannot fully mediate. Key threats include full-disk encryption bypass via cold boot attack (DRAM memory remanence — capturing encryption keys from powered-down or hibernated memory), evil maid attacks enabling firmware modification or hardware keylogger implantation when a device is left unattended, direct memory access (DMA) attacks via Thunderbolt and PCIe interfaces (enabling read/write of any physical memory address with no OS permission check), USB-based BadUSB attacks emulating trusted HID devices (keyboards, network adapters) to execute payloads without user interaction, rogue peripheral implantation (hardware keyloggers, malicious charging cables with integrated compute), and theft of an unlocked or unencrypted device granting direct file system access. Financial sector precedents include trading floor physical security incidents, overnight device compromise during international travel by bank employees, and physical access to Bloomberg terminals or treasury system workstations outside supervised hours.

### ZTA Integration

Layer 2 physical controls operate at the lowest enforcement layer in the NIST SP 800-207 Data Plane, providing the hardware-level assurance that underpins all higher-layer controls. Critically, physical controls are the only category where enforcement precedes and is independent of the Policy Enforcement Plane — a stolen device with BitLocker TPM+PIN (PH-01) or an Apple Silicon endpoint with Secure Enclave memory encryption (PH-05) enforces data confidentiality without any cloud connectivity, network check-in, or PDP evaluation.

**Trust Signal Generation (PDP inputs):** PH-01 and PH-02 (full disk encryption state) are surfaced as MDM compliance attributes feeding the Policy Engine via Intune compliance reporting. Kernel DMA Protection status (PH-03) is similarly reported as a compliance attribute. FileVault escrow key receipt (Jamf/Intune) confirms encryption is active and the key is recoverable. These signals contribute to the device trust score evaluated by Conditional Access — an endpoint reporting missing FDE generates a Hard gate compliance failure and is denied access to all regulated resources.

**Local Policy Enforcement (PEPs):** PH-06 (USB block), PH-07 (HID allowlisting), PH-12 (screen lock), and PH-08 (memory overwrite) are endpoint-resident PEPs operating independent of cloud connectivity. Their enforcement does not require a live management channel — the MDM-pushed policy remains active whether the device is online or offline. This local enforcement posture is the appropriate design for physical security controls, which must remain effective when the device is in an adversarial physical environment where network access may be maliciously controlled.

**Policy Administration (PAP):** PH-10 and PH-11 (remote wipe), PH-04 (Thunderbolt authorisation), and PH-13 (privacy filter mandate) operate in the Policy Administration layer — they are governance and configuration controls that define the physical security posture rather than enforcing it at runtime. Remote wipe requires PA-layer actions triggered by a security event; Thunderbolt authorisation policy is deployed via MDM as a PA function that configures the local PEP.

**Continuous Monitoring (Management Plane):** PH-14 (chassis intrusion) generates physical tamper events that flow to the SIEM. Remote wipe audit logs, USB insertion events (regardless of block outcome), and Thunderbolt device connection events all contribute to the continuous monitoring posture — providing the physical access telemetry that the Policy Engine uses to assess device trust between MDM check-in cycles.
